Introduction

This API provides access functions to query what documents are available within a patient record, where the document resides within the logical structure of the patient record and to retrieve the individual documents. It also provides access to limited patient demographic information and the ability to search for a patient but the patient data should not be considered to be authoritative as the CPF system only maintains a limited and potentially out of date copy of this data. The PAS systems should be considered the authoritative source of patient and episode details.

The API is supported using a REST interface with returned values in xml or json formats.

The API will require correct authentication against a current CPF enabled user account before any access to the patient record is granted. The access to particular records is determined by the user account settings and restrictions. All access to the patient records via this API will be fully logged in the CPF system's audit log.

The API provides access to a logical model of the patient record and allows simple navigation of that model and retrieval of documents stored within that model.

Logical Model

Patient
This represents individual patients and a limited set of the patient details sufficient to uniquely identify that patient. It is not intended to supply full patient details as these are available using the same PatientId from other systems. Note that the PatientId is usually made up of two components, the Namespace and the URNO. A patient may have multiple IDs in the system (aliases) but one of these is the master ID for that patient and this is the one that is visible in this object. Queries for patients using one of the other PatientIds will return the patient but the return results will contain the primary ID.
Section
A section is a container for either documents or for episodes (an episode generally represents a patient attendance at a hospital or clinic). Sections have names to uniquely identify them. Sections are like physical folders. They contain either episodes or documents (but not both).
Episode
An episode is a container for documents. These documents generally relate to an admission or clinic that a patient attended. Episodes are like physical folders containing the documents created and pertinent to a particular admission for a patient.
Document
A document is one or more pages of the patient record. It has a document type identified by a document code. It has a mime type which identifies the format of the document and indicates how it should be interpreted.
DocumentData
The binary data that is the document. The type of this data is given by the mime type field in the containing Document object. There is only one DocumentData for each Document object.

REST Resources

This API supports a Representational State Transfer (REST) model for accessing a set of resources through a fixed set of operations. The following resources are accessible through the RESTful model:

  • GetActivityFeed
  • GetAlerts
  • GetDocument
  • GetEpisode
  • GetPatient
  • GetPatientList
  • GetSearchList
  • GetSection
  • SearchDocuments
  • SearchEpisodes
  • SearchPatients
  • SearchSections
  • UpdateMyPatients

The REST resources expose a data model that is supported by a set of client-side libraries that are made available on the files and libraries page.

There is also a WADL document describing the REST API.

Data Model

All endpoints act on a common set of data. The data can be represented with difference media (i.e. "MIME") types, depending on the endpoint that consumes and/or produces the data. The data can described by XML Schema, which definitively describes the XML representation of the data, but is also useful for describing the other formats of the data, such as JSON.

This document will describe the data using terms based on XML Schema. Data can be grouped by namespace, with a schema document describing the elements and types of the namespace. Generally speaking, types define the structure of the data and elements are instances of a type. For example, elements are usually produced by (or consumed by) a REST endpoint, and the structure of each element is described by its type.
